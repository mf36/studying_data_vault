The problem PIT tables solve
1
Multiple satellites per Hub update independently
HUB_CUSTOMER has SAT_CUST_DETAILS and SAT_CUST_CONTACT. Each has its own timeline. A name change doesn't create a new contact row, and vice versa.
SAT_CUST_DETAILS + SAT_CUST_CONTACT
2
Joining them at a point-in-time is painful
Every satellite needs a correlated subquery to find "most recent row ≤ snapshot date". With 4+ satellites this becomes deeply nested and slow.
One correlated subquery per satellite
3
PIT pre-computes the join keys
One row per customer × snapshot date. Each satellite gets a column: its effective LOAD_DTS at that moment. Downstream: simple equi-joins, no subqueries.
O(1) per satellite join
Without PIT — correlated subquery per satelliteBEFORE
-- "What did we know about each customer on 2024-01-15?"
SELECT h.CUSTOMER_ID, d.FULL_NAME, d.EMAIL, c.PHONE
FROM HUB_CUSTOMER h
LEFT JOIN SAT_CUST_DETAILS d
  ON  d.CUSTOMER_HK = h.CUSTOMER_HK
 AND  d.LOAD_DTS = (SELECT MAX(d2.LOAD_DTS) FROM SAT_CUST_DETAILS d2
                     WHERE d2.CUSTOMER_HK = h.CUSTOMER_HK
                       AND d2.LOAD_DTS <= '2024-01-15')
LEFT JOIN SAT_CUST_CONTACT c
  ON  c.CUSTOMER_HK = h.CUSTOMER_HK
 AND  c.LOAD_DTS = (SELECT MAX(c2.LOAD_DTS) FROM SAT_CUST_CONTACT c2
                     WHERE c2.CUSTOMER_HK = h.CUSTOMER_HK
                       AND c2.LOAD_DTS <= '2024-01-15');
-- 6 satellites = 6 correlated subqueries. Painful.
Build PIT_CUSTOMER — run on a daily scheduleBUILD
WITH snapshots AS (
  SELECT h.CUSTOMER_HK, s.SNAPSHOT_DTS
  FROM HUB_CUSTOMER h
  CROSS JOIN (SELECT dt::TIMESTAMP AS SNAPSHOT_DTS FROM DATE_SPINE) s
)
INSERT INTO PIT_CUSTOMER (CUSTOMER_HK, SNAPSHOT_DTS, SAT_DETAILS_LDTS, SAT_CONTACT_LDTS)
SELECT
  sn.CUSTOMER_HK, sn.SNAPSHOT_DTS,
  MAX(d.LOAD_DTS) FILTER(WHERE d.LOAD_DTS <= sn.SNAPSHOT_DTS) AS SAT_DETAILS_LDTS,
  MAX(c.LOAD_DTS) FILTER(WHERE c.LOAD_DTS <= sn.SNAPSHOT_DTS) AS SAT_CONTACT_LDTS
FROM snapshots sn
LEFT JOIN SAT_CUST_DETAILS  d ON d.CUSTOMER_HK = sn.CUSTOMER_HK
LEFT JOIN SAT_CUST_CONTACT  c ON c.CUSTOMER_HK = sn.CUSTOMER_HK
GROUP BY sn.CUSTOMER_HK, sn.SNAPSHOT_DTS;

With PIT — clean equi-joins, no subqueriesAFTER
SELECT h.CUSTOMER_ID, d.FULL_NAME, d.EMAIL, c.PHONE, c.COUNTRY_CODE
FROM PIT_CUSTOMER pit
JOIN      HUB_CUSTOMER    h  ON h.CUSTOMER_HK = pit.CUSTOMER_HK
LEFT JOIN SAT_CUST_DETAILS d  ON d.CUSTOMER_HK = pit.CUSTOMER_HK
                              AND d.LOAD_DTS    = pit.SAT_DETAILS_LDTS
LEFT JOIN SAT_CUST_CONTACT c  ON c.CUSTOMER_HK = pit.CUSTOMER_HK
                              AND c.LOAD_DTS    = pit.SAT_CONTACT_LDTS
WHERE pit.SNAPSHOT_DTS = '2024-01-15';
-- Works identically for 10 satellites — just add 10 equi-joins.
What is the Business Vault?

The Business Vault sits on top of Raw Vault. It applies business rules, calculations, and derived logic. Still insert-only and auditable — but speaks the language of the business, not the source system.

Components
bSAT
Business Satellite
Derived attributes computed from raw vault. E.g., CUSTOMER_SEGMENT from order history. Same structure as raw sat — but populated by a business rule, not a source feed.
SAL
Same-As Link
Links two hub keys that represent the same real-world entity across different systems. Used for identity resolution between CRM and ERP customer IDs.
xRef
Cross-Reference
Maps business keys across source systems into a unified key namespace. Complements SALs for master data management scenarios.
BRDG
Bridge Table
Like a PIT for Links — pre-computes multi-hop relationship joins. E.g., Customer → Order → Product flattened for fast mart queries.
BSAT_CUST_SEGMENT — deriving customer tier from order historybSAT
INSERT INTO BSAT_CUST_SEGMENT (CUSTOMER_HK,LOAD_DTS,SEGMENT,LIFETIME_VALUE,HASH_DIFF,RULE_VERSION)
WITH totals AS (
  SELECT l.CUSTOMER_HK, SUM(s.TOTAL_AMT) AS LTV
  FROM  LNK_CUST_ORDER l
  JOIN  SAT_ORDER_STATUS s ON s.ORDER_HK = l.ORDER_HK
   AND  s.LOAD_DTS = (SELECT MAX(x.LOAD_DTS) FROM SAT_ORDER_STATUS x WHERE x.ORDER_HK=l.ORDER_HK)
  GROUP BY l.CUSTOMER_HK
)
SELECT
  CUSTOMER_HK, CURRENT_TIMESTAMP,
  CASE WHEN LTV >= 10000 THEN 'PLATINUM'
       WHEN LTV >= 1000  THEN 'GOLD'
       WHEN LTV >= 100   THEN 'SILVER'
       ELSE 'BRONZE' END,
  LTV, MD5(CONCAT(CUSTOMER_HK,LTV)), 'v1.2'
FROM totals t
WHERE NOT EXISTS (SELECT 1 FROM BSAT_CUST_SEGMENT b
  WHERE b.CUSTOMER_HK=t.CUSTOMER_HK AND b.HASH_DIFF=MD5(CONCAT(t.CUSTOMER_HK,t.LTV)));
SAL_CUSTOMER — identity resolution across CRM and ERPSAL
INSERT INTO SAL_CUSTOMER (SAL_HK,CUSTOMER_HK_1,CUSTOMER_HK_2,LOAD_DTS,REC_SRC,CONFIDENCE)
SELECT
  MD5(CONCAT(MD5(m.CRM_ID),'||',MD5(m.ERP_ID))),
  MD5(m.CRM_ID), MD5(m.ERP_ID),
  CURRENT_TIMESTAMP, 'MDM_SYSTEM', m.CONFIDENCE_SCORE
FROM mdm_match_results m
WHERE NOT EXISTS (SELECT 1 FROM SAL_CUSTOMER s
  WHERE s.SAL_HK = MD5(CONCAT(MD5(m.CRM_ID),'||',MD5(m.ERP_ID))));
DIM_CUSTOMER — Info Mart view for BI toolsMART
CREATE OR REPLACE VIEW DIM_CUSTOMER AS
SELECT
  h.CUSTOMER_ID, d.FULL_NAME, d.EMAIL, c.PHONE,
  r.COUNTRY_NAME, r.REGION,
  seg.SEGMENT AS CUSTOMER_SEGMENT, seg.LIFETIME_VALUE
FROM HUB_CUSTOMER h
JOIN PIT_CUSTOMER pit ON pit.CUSTOMER_HK = h.CUSTOMER_HK
  AND pit.SNAPSHOT_DTS = (SELECT MAX(p2.SNAPSHOT_DTS) FROM PIT_CUSTOMER p2 WHERE p2.CUSTOMER_HK=h.CUSTOMER_HK)
LEFT JOIN SAT_CUST_DETAILS  d   ON d.CUSTOMER_HK=pit.CUSTOMER_HK AND d.LOAD_DTS=pit.SAT_DETAILS_LDTS
LEFT JOIN SAT_CUST_CONTACT  c   ON c.CUSTOMER_HK=pit.CUSTOMER_HK AND c.LOAD_DTS=pit.SAT_CONTACT_LDTS
LEFT JOIN BSAT_CUST_SEGMENT seg ON seg.CUSTOMER_HK=h.CUSTOMER_HK
  AND seg.LOAD_DTS=(SELECT MAX(s2.LOAD_DTS) FROM BSAT_CUST_SEGMENT s2 WHERE s2.CUSTOMER_HK=h.CUSTOMER_HK)
LEFT JOIN REF_COUNTRY r ON r.COUNTRY_CODE=c.COUNTRY_CODE;

Reading this honestly: Data Vault carries a real structural tax — roughly 3–5× tables, 2–3× load compute, 2–4× ETL steps compared to an equivalent star schema. This overhead is concentrated in the load layer and raw vault query layer. The info mart + PIT architecture is specifically designed to recover that cost at query time.

The break-even point is typically around 15–25 source entities. Below that, a well-designed star schema is simpler. Above it — especially with multiple source systems, regulatory audit requirements, or frequent schema changes — Data Vault's structural properties increasingly justify the overhead.

The honest take: Data Vault is a platform investment, not a shortcut. It pays back through flexibility, auditability, and parallelism — but only if your project lives long enough and scales wide enough to recoup the upfront complexity. If you're building a dashboard for one team from one source and you'll know what questions they'll ask, a star schema will ship faster and be easier to maintain. Data Vault earns its keep when the alternative is a spaghetti of point-to-point integrations, monthly ALTER TABLE emergencies, or an audit failure.
